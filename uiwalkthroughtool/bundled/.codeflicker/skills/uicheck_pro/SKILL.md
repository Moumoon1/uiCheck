# Goal

1. 找出可稳定确认的正式问题
2. 记录疑似问题
3. 文档输出为表格
4. 双图必须框同一问题对象
5. 图片必须先保存到 workspace

## Reference 文件结构

SKILL.md 放总规则和最高优先级约束，reference 按阶段拆分：

| 文件 | 加载阶段 | 内容 |
|------|---------|------|
| `issue_rules.md` | analysis | 问题分类、优先关注、设计完整性、数据排除 |

# Input Identity & Read Verification

- `dev` = 开发实现图，`design` = 设计目标图
- 用户已指定 dev/design 时不得交换；未指定时才可结合文件名与上下文推断
- `@绝对路径` 只负责把本地图片作为附件传入；是否能真正读图取决于当前运行模型是否具备 vision 能力
- 若当前模型是纯文本模型或未开启 vision，必须直接视为“无法读图”，不得因为 prompt 中出现了 `@绝对路径` 就声称已经看到了图片
- 开始判断前，必须先逐一读取两张原图，确认：
  - 图片已成功读取
  - 能区分 dev / design 身份
  - 能看清主要模块与关键元素
- 若任一图片未读到、读错、路径失效、内容不可辨认，必须立即停止并输出：`读图验证失败：[原因]`
- 未完成读图验证前，不得输出正式问题、疑似问题或最终结论
- 分析过程中必须反复保持身份不变：`dev = 已实现结果`，`design = 应还原目标`

# Runtime Guardrails（最高优先级约束）

- 图片输入必须使用 `@绝对路径` 格式；普通文本路径不会自动当作图片读取
- `@绝对路径` 只是附件引用格式，不等于模型一定具备读图能力
- 若当前模型不支持 vision，则应把图片读取视为失败，停止分析，不得宣称模型已读取图片
- 真实 dev/design 图片必须用 `@绝对路径` 放进 prompt，且优先使用独立行块格式，不要把 `@图片路径` 混在自然语言句子里，例如：
  - `开发稿：\n@/absolute/path/dev.png`
  - `设计稿：\n@/absolute/path/design.png`
- 分析用中间图片必须优先落到 workspace 内可读目录（如 `runtime_images/`），不要默认写到 `/tmp`，避免 IDE Agent / CLI 无权限或不可见
- 在调用模型前必须打印并落盘最终 prompt 与最终图片 refs，确认传入的确实是本次运行的 `@绝对路径`
- 对 `/api/analyze/uicheck` 主链路，优先调试 `claude2 + stream-json + step2AnalysisPrompt`；旧的 text/fullPrompt 流程只作为 legacy/debug 旁路
- 当怀疑 CLI 没有真正附图时，先在终端做最小验证：直接用 `codeflicker -q` + 本次图片 `@绝对路径`，要求模型回答图片里真实可见的标题文字、主色、顶部模块；若能回答具体可见内容，说明当前环境下 `-q` 可读图，问题更可能在主链路 prompt 或验证门禁
- 背景信息传纯文本内容，不要把本地文件路径当作背景正文
- 结论只能来自当次截图中可见证据；截图里看不到的模块名、业务名、文案，不得凭 reference 补写
- 参考文件只提供字段说明、硬约束、排除项和中性模板；不得把 reference 中的业务名、模块名、示例句子直接当作本次结论

> 详细运行时规则见 `reference/runtime_guardrails.md`

# Issue Rules

差异分三类：

- 正式问题
- 疑似问题
- 不纳入问题（动态数据 / 滚动 / 状态栏 / 轻微渲染）

优先关注：
- 模块缺失 / 顺序错误
- 骨架 / 层级 / 对齐问题
- 关键组件 / 颜色 / 尺寸
- 信息架构错误
- 标签 / 图例 / 字段规则错误

---

## Design Completeness Rule

必须以设计稿为基准逐模块检查：

- 设计稿中每个模块 / 元素，都必须在开发页确认是否存在
- 若存在于设计稿但开发页不存在 → 必须输出正式问题
- 不得因为未发现差异而跳过模块

---

## Data & Chart Exclusion Rule

以下差异默认不作为问题：

- 金额、数量、百分比、转化率
- 日期、时间、用户名、头像、排名、状态
- 图表数据值、报表数值
- 数据驱动的图形变化（宽度 / 高度 / 面积 / 曲线）

对于图表 / 漏斗：

只检查：
- 结构
- 标签
- 单位
- 图例
- 样式 / 布局 / 层级

不检查：
- 数值一致性
- 图形形态（漏斗宽度、柱高等）

只有以下情况才算问题：

- 字段缺失
- 标签错误
- 单位错误
- 图例错误
- 结构 / 样式 / 层级错误
- 设计稿明确固定形态但未实现

# Screenshot Rules

## Assets Screenshot Guide（按需加载）

assets/screenshots 示例图用于学习截图框选规范：
- `good_pair_annotation.png` — 双图必须框同一对象
- `good_element_level_box.png` — 元素级精确框选
- `good_module_level_box.png` — 模块级保留上下文
- `bad_pair_annotation.png` — 错误配对、错误投影和错误框法

**加载策略**：
- analysis 阶段默认不加载 assets/examples（减少幻觉污染）
- 截图规则调试或误判案例排查时按需加载
- 截图/docx 阶段复用同一份 issues JSON，由本地程序裁图、画框和导出

## CropRegion vs Box

坐标输出必须拆分为两组：

- **CropRegion**：裁图区域，略大于问题区域，保留上下文，方便用户看懂问题所处环境
- **Box**：红框区域，问题元素本身的精确位置，框尽量小而准

规则：
- 元素级问题 → Box 尽量小，只框具体元素
- 模块级问题 → Box 框住整个出问题的模块
- 细节级问题 → Box 框住对应细节区域
- `Box == CropRegion` 只适用于整个裁图区域都是问题的大范围结构问题
- 不合格的 Box：框整张图（`top=0 bottom=1 left=0 right=1`）→ 必须缩小

## Element Matching Rule

双图框选必须基于“元素匹配”，不得基于位置：

1. 先在设计稿识别元素（按钮 / 卡片 / 图表等）
2. 在开发页按语义查找对应元素
3. 找到后再框选

## No Position Projection Rule

禁止：
- 把设计稿框直接映射到开发图同一位置

原因：
- 尺寸不同
- 布局偏移
- 滚动差异

## Matching Failure Handling

若开发页找不到对应元素：

- 不得框空白
- 不得猜位置
- 必须标记为“组件缺失”或“未实现”

截图方式：
- 设计稿：精确框元素
- 开发页：框模块区域或标注未实现区域

## 基础规则

- 两边必须框同一对象
- 框选必须与问题描述一致
- 一条问题必须有双图
- 不得用模糊区域代替

# Problem Description Rule

只允许：

- 问题
- 建议

不允许：

- 标题
- 位置
- 长分析
- reference 示例原句
- 截图里不存在的业务词

写法要求：

- 只写当前截图可见差异
- 句子必须可执行、可复核
- 不能因为 reference 出现过某个说法，就直接套到本次问题里

---

# Reference Usage Rule

- reference 的作用是补充边界、排除项、字段规范和中性模板
- assets 的作用是提供截图规范示例，尤其用于指导 `CropRegion` / `Box` 的框法
- reference 中的示例只能当“句式模板”，不能当“内容模板”
- 若示例里出现具体业务对象、模块名或视觉细节，输出时必须改写成基于当前截图证据的描述
- 如果无法从截图证据中复核某个说法，就不要输出该说法

---

# Document Output Rules

结构固定：

1. 标题
2. 摘要
3. 页面总览
4. 问题表格

表头：

- 序号 / 问题描述 / 状态 / 截图 / 日期 / 优先级 / 走查人 / 跟进人

---

# Folder Mapping Rule

- 按文件名匹配 dev / design
- 不匹配的必须提示
- 每一对独立走查

---

# Restrictions

- 不得交换 dev/design
- 不得在未读图验证通过前输出结论
- 不得把动态数据当问题
- 不得把数据驱动图形当问题
- 不得混淆正式 / 疑似
- 不得框错对象
- 不得框空白
- 不得用位置映射代替元素匹配
- 不得把“数据差异”脑补为“规则问题”
- 不得复述 reference 示例中的现成问题句子
