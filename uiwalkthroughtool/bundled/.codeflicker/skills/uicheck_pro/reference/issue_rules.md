# Issue Rules

## 正式问题
正式问题要满足：
- 差异明确稳定
- 不太可能由截图条件造成
- 可以直接提给开发修改

常见正式问题类型：
- 模块类型不一致
- 核心信息结构不一致
- 对齐关系明显错误
- 关键按钮大小、颜色或位置明显不对
- 核心区域被遮挡、压叠
- 模块顺序明显错误
- 切图/裁剪错误（卡片或图片边缘被截断、元素溢出被裁掉、圆角被切掉）

## 疑似问题
疑似问题用于记录：
- 看起来有偏差
- 对设计质量重要
- 但暂时不适合直接作为“必须修改”的正式问题

常见疑似问题类型：
- 曲线形态疑似更硬
- 光效疑似过重
- 背景层次疑似更弱
- 圆角气质疑似不同
- 局部留白节奏疑似不同

## 问题描述怎么写

### 正式问题
只写：
- 问题
- 建议

要求：
- 使用当前截图中可见、可复核的描述
- 描述差异本身，不复述 reference 里的现成业务对象
- 句子要能直接指导修改

### 疑似问题
只写：
- 问题
- 建议

要求：
- 明确这是“疑似”而非已确认事实
- 建议里写清如何进一步确认
- 仍然只能基于截图可见证据，不得凭示例补内容

## 不要这样写
- 写成长段小作文
- 在问题描述里重复写位置
- 在问题描述里重复写优先级和状态
- 在问题描述里塞很多背景解释
- 直接复制示例句子作为本次结论
