# Claude Code & Codex 使用指南

## 🎯 核心概念

这个 UI 走查工具是一个**本地 Web 服务**，通过浏览器使用。Claude Code 和 Codex 的作用是：
- **自动启动服务器**（无需手动命令）
- **解释使用方法**（作为智能助手）
- **帮助理解结果**（解读走查报告）

## 📋 详细使用流程

### 一、Claude Code 用户

#### 1️⃣ 启动服务

**方式一：对话启动（推荐）**
```
用户: 启动设计师平台
或: 打开走查工具
或: 我要用 UI 走查
```

Claude Code 会自动：
```bash
cd designer-platform
npm install  # 自动检查并安装依赖
node server.js  # 启动服务器
```

**方式二：手动启动**
```bash
cd claudeprog/designer-platform
npm start
```

#### 2️⃣ 访问界面

服务启动后，Claude Code 会告知：
```
✅ 服务器已启动
🌐 访问地址: http://localhost:3000
```

打开浏览器访问上述地址，会看到走查界面。

#### 3️⃣ 网页选项说明

**A. 页面类型选择**
```
┌─────────────────────────────────┐
│ 📄 页面类型                      │
│ [C端页面] [B端页面]              │
└─────────────────────────────────┘
```

- **C端页面**（默认）：
  - 适用：移动端活动页、功能页
  - 特点：关注视觉还原和结构一致性
  - 分析规则：严格对比颜色、间距、字体

- **B端页面**：
  - 适用：管理后台、数据看板
  - 特点：关注功能完整性和交互逻辑
  - 分析规则：更宽松，允许合理的设计调整

**B. 上传模式**

```
┌─────────────────────────────────┐
│ 🖥️ 页面上传                      │
│ [📷 单页面上传] [📁 文件夹上传]   │
└─────────────────────────────────┘
```

**单页面上传**（推荐）：
```
┌──────────────┐  ┌──────────────┐
│  🖥️ 开发长截图  │  │  🎨 设计稿长图  │
│              │  │              │
│   点击上传    │  │   点击上传    │
│              │  │              │
└──────────────┘  └──────────────┘
```

- 适合：单个页面的走查
- 命名建议：`dev_xxx.png` 和 `design_xxx.png`
- 最多支持 10 张图片

**文件夹上传**：
```
┌──────────────┐  ┌──────────────┐
│ 📁 开发截图    │  │ 📁 设计稿      │
│    文件夹      │  │    文件夹      │
│              │  │              │
│   点击选择    │  │   点击选择    │
└──────────────┘  └──────────────┘
```

- 适合：批量走查多个页面
- **关键**：开发截图和设计稿文件名必须相同
- 示例：
  ```
  开发截图文件夹/       设计稿文件夹/
  ├── 首页.png    →    ├── 首页.png
  ├── 列表页.png  →    ├── 列表页.png
  └── 详情页.png  →    └── 详情页.png
  ```

**C. 背景信息（可选）**
```
┌─────────────────────────────────┐
│ 📝 背景信息（可选）               │
│                                 │
│ 补充走查背景，例如：本次重点检查  │
│ 首页改版，关注顶部导航和底部 CTA  │
│ 按钮的还原度...                  │
└─────────────────────────────────┘
```

- 作用：帮助 AI 理解走查重点
- 示例：
  - "本次是 618 活动页，重点关注促销模块"
  - "新版本改版，检查导航栏和底部 Tab"
  - "B端管理后台，关注表格和筛选功能"

**D. 视觉模型选择**

```
┌─────────────────────────────────┐
│ 🤖 视觉模型                      │
│ ┌─────────────────────────────┐ │
│ │ Kimi K2.5               ▼   │ │
│ └─────────────────────────────┘ │
│                                 │
│ ○ Kimi K2.5                     │
│ ○ Claude Sonnet 4.6  [额度受限] │
│ ● GPT 5.4            [推荐]    │
│ ○ Gemini 3.1 Pro     [额度受限] │
└─────────────────────────────────┘
```

**模型对比**：

| 模型 | 代号 | 特点 | 推荐场景 | 注意事项 |
|------|------|------|----------|----------|
| **Kimi K2.5** | `kimi-k2.5` | 中文理解强，速度快 | 中文页面、快速走查 | 国产模型，稳定可靠 |
| **Claude Sonnet 4.6** | `claude` | 视觉理解最强 | 复杂页面、细节对比 | 有额度限制（约 100 次/天） |
| **GPT 5.4** | `5` | 综合能力强（推荐） | 所有场景 | 推荐首选，平衡性能和额度 |
| **Gemini 3.1 Pro** | `gemini` | 多模态能力强 | 多语言页面 | 有额度限制 |

**如何选择**：
1. **首次使用**：选择 `GPT 5.4`（推荐）
2. **中文页面**：选择 `Kimi K2.5`
3. **复杂细节**：选择 `Claude Sonnet 4.6`（注意额度）
4. **额度用完**：切换其他模型

#### 4️⃣ 开始走查

点击"开始走查"按钮后：

```
┌─────────────────────────────────┐
│ ⏳ 处理中...                     │
│                                 │
│ ✅ 上传文件                      │
│ ⏳ CodeFlicker 分析中...         │
│ ○ 分析完成                      │
│                                 │
│ 正在分析设计稿结构...            │
└─────────────────────────────────┘
```

**处理流程**：
1. **Step 1**：分析设计稿结构
   - 识别页面模块
   - 提取视觉特征
   - 生成模块清单

2. **Step 2**：对比开发稿
   - 逐个模块对比
   - 识别差异
   - 标注问题位置

3. **Step 3**：生成报告
   - 分类问题（确认/疑似）
   - 生成带标注的截图
   - 输出修复建议

#### 5️⃣ 查看结果

**结果结构**：

```
📋 走查结果
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【确认问题】（必须修复）
┌──────────────────────────────┐
│ #1 标题字号偏小               │
│ 问题：开发稿标题 16px，设计稿 18px │
│ 建议：调整标题字号至 18px      │
│ 优先级：P1                    │
│ 位置：顶部导航                │
│ [开发截图] [设计截图]         │
└──────────────────────────────┘

【疑似问题】（需人工确认）
┌──────────────────────────────┐
│ #2 按钮圆角可能不一致         │
│ 问题：开发稿圆角 4px，设计稿 6px │
│ 理由：可能是设计调整          │
│ 建议：与设计师确认            │
│ 优先级：P2                    │
│ [开发截图] [设计截图]         │
└──────────────────────────────┘
```

**问题优先级**：
- **P0**：严重问题，必须修复（如功能缺失、严重错位）
- **P1**：重要问题，建议修复（如颜色、字号差异）
- **P2**：次要问题，可选修复（如微小的间距差异）

## 💡 最佳实践

### 1. 截图准备

**开发稿截图**：
- 使用 Chrome 开发者工具的 "Capture full size screenshot"
- 或使用浏览器扩展（如 Awesome Screenshot）
- 确保截图完整，不要遗漏底部内容

**设计稿截图**：
- 从 Figma/Sketch 导出完整设计稿
- 导出分辨率与开发稿一致
- 保持原始颜色，不要压缩

### 2. 命名规范

```
推荐命名：
✅ dev_首页.png
✅ design_首页.png
✅ dev_列表页_v2.png
✅ design_列表页_v2.png

不推荐：
❌ 截图1.png
❌ 设计稿.png
❌ 未命名.png
```

### 3. 背景信息

**好的背景信息示例**：
```
本次是 618 活动页改版，重点关注：
1. 顶部促销 Banner 的动画效果
2. 商品卡片的布局和间距
3. 底部"立即购买"按钮的位置
设计稿版本：v2.3，已确认最终设计
```

**不好的背景信息**：
```
帮我检查一下
```

### 4. 结果使用

**确认问题**：
- 直接按建议修复
- 优先处理 P0 和 P1 问题
- 修复后重新走查验证

**疑似问题**：
- 与设计师沟通确认
- 可能是合理的设计调整
- 确认后再决定是否修复

## ⚠️ 常见问题

### Q1: 服务启动失败

```
问题: npm install 报错

解决:
1. 检查 Node.js 版本：node -v（需要 >= 18）
2. 清理缓存：npm cache clean --force
3. 删除 node_modules：rm -rf node_modules
4. 重新安装：npm install
```

### Q2: 模型额度用完

```
问题: Claude 模型提示额度用完

解决:
1. 切换其他模型（GPT 5.4 / Kimi K2.5）
2. 等待额度刷新（通常 12 小时）
3. 检查 API 密钥配置
```

### Q3: 截图无法识别

```
问题: AI 无法识别图片内容

解决:
1. 检查图片格式（支持 PNG/JPG/WebP）
2. 确保图片大小 < 20MB
3. 避免截图过于模糊
4. 尝试切换其他模型
```

### Q4: 结果不准确

```
问题: 走查结果有误报或漏报

解决:
1. 优化背景信息，说明走查重点
2. 切换更强大的模型（如 Claude）
3. 确保截图完整、清晰
4. 对于疑似问题，人工二次确认
```

## 📚 进阶用法

### 1. 批量走查

```

建议使用文件夹上传模式：

1. 准备两个文件夹：
   - dev_screenshots/
   - design_screenshots/

2. 确保文件名相同：
   dev_screenshots/     design_screenshots/
   ├── 首页.png    →    ├── 首页.png
   ├── 列表.png    →    ├── 列表.png
   └── ...              └── ...

3. 在界面选择"文件夹上传"
4. 分别选择两个文件夹
5. 系统会自动配对并批量走查

需要我帮您准备文件结构吗？
```

### 2. 定期回归

```
用户: 每周都要走查，怎么保持高效？

建议:
1. 创建固定的文件夹结构
2. 使用命名规范（如 v1.0_dev_首页.png）
3. 保存常用的背景信息模板
4. 固定使用一个模型，熟悉其特点
5. 建立问题库，避免重复报告

我可以帮您：
- 创建走查清单
- 生成背景信息模板
- 整理历史问题
```

### 3. 团队协作

```
用户: 团队多人使用，怎么避免冲突？

建议:
1. 每人使用不同的端口（修改 .env 中的 PORT）
2. 定期清理 inputs/ 和 outputs/ 目录
3. 使用 Git 管理配置和规则
4. 建立团队的问题标注规范
5. 共享最佳实践和模型选择经验

团队使用技巧：
- 指定专人维护规则文件

```
