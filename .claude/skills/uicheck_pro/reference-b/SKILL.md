# Goal

1. 找出B端页面中可稳定确认的正式问题
2. 记录疑似问题
3. 文档输出为表格
4. 双图必须框同一问题对象（双图坐标独立定位，禁止位置投影）
5. 图片必须先保存到 workspace

# Input Identity & Read Verification

- `dev` = 开发实现图，`design` = 设计目标图
- 用户已指定 dev/design 时不得交换；未指定时才可结合文件名与上下文推断
- `@绝对路径` 只负责把本地图片作为附件传入；是否能真正读图取决于当前运行模型是否具备 vision 能力
- 若当前模型是纯文本模型或未开启 vision，必须直接视为"无法读图"，不得因为 prompt 中出现了 `@绝对路径` 就声称已经看到了图片
- 开始判断前，必须先逐一读取两张原图，确认：
  - 图片已成功读取
  - 能区分 dev / design 身份
  - 能看清主要模块与关键元素
- 若任一图片未读到、读错、路径失效、内容不可辨认，必须立即停止并输出：`读图验证失败：[原因]`
- 未完成读图验证前，不得输出正式问题、疑似问题或最终结论
- 分析过程中必须反复保持身份不变：`dev = 已实现结果`，`design = 应还原目标`

# Runtime Guardrails（最高优先级约束）

- 图片输入必须使用 `@绝对路径` 格式；普通文本路径不会自动当作图片读取
- `@绝对路径` 只是附件引用格式，不等于模型一定具备读图能力
- 若当前模型不支持 vision，则应把图片读取视为失败，停止分析，不得宣称模型已读取图片
- 真实 dev/design 图片必须用 `@绝对路径` 放进 prompt，且优先使用独立行块格式
- 分析用中间图片必须优先落到 workspace 内可读目录
- 在调用模型前必须打印并落盘最终 prompt 与最终图片 refs
- 背景信息传纯文本内容，不要把本地文件路径当作背景正文
- 结论只能来自当次截图中可见证据
- 参考文件只提供字段说明、硬约束、排除项和中性模板

> 详细运行时规则见 `reference/runtime_guardrails.md`

# Issue Rules（B端）精准描述总纲

B端走查问题分四类，每类都有强制描述格式（详见 `reference-b/issue_rules.md`）：

## 1. 枚举 / 列表类
Tab、表格列、操作按钮、筛选控件、面包屑、单选/多选选项、下拉选项
→ **只需指出缺少的、多余的或错误的具体项名，不需要把两侧所有项都列出来**
→ 不合格：`Tab 项不一致` / `列缺失` / `按钮缺失`
→ 合格：`开发稿 Tab 缺少"审核中"` / `开发稿缺少"金额"列` / `开发稿操作列缺少"详情"按钮`

## 2. 组件样式类
按钮、图标、输入框、标签/Badge、下拉框外观、单选控件、多选控件
→ **必须描述两侧各自的具体样式（填充/边框/颜色/形状/有无图标等）**
→ 不合格：`按钮样式不对` / `输入框不对`
→ 合格：`设计稿主按钮为实色蓝色填充+白字，开发稿为边框线框+蓝字`

## 3. 量化类
间距、行高、字号、字重、信息层级、对齐方式
→ **必须给出估值或相对比较，禁止只说"偏大/偏小"**
→ 不合格：`间距偏小` / `字号不同`
→ 合格：`设计稿表格行高约 48px，开发稿约 36px，偏小约 25%`

## 4. 图表内部结构类
折线样式、数据点、网格线、坐标轴、图例、填充、柱体细节、参考线、数据标注
→ **图表外形一致不代表内部结构一致，必须逐项检查内部细节**
→ 不合格：只报"图表外形和类型一致" / 不报图表内部差异
→ 合格：`设计稿折线为虚线，开发稿为实线` / `设计稿 Y 轴有单位"万元"，开发稿无单位`

## CropRegion vs Box

坐标输出必须拆分为两组：

- **CropRegion**：裁图区域，略大于问题区域，精确到模块级（不是整页）
- **Box**：红框区域，问题元素本身的精确位置，精确到具体元素（不是整模块）

关键规则：
- **devCropRegion 和 designCropRegion 必须完全相同**（保证用户左右对比对齐）
- **devBox 和 designBox 坐标必须不同**（B端布局差异大，必须各自独立定位）
- 不合格的 Box：框整张图（top=0 bottom=1 left=0 right=1）→ 必须缩小

## Element Matching Rule

双图框选必须基于"元素匹配"，不得基于位置：

1. 先在设计稿识别元素（表格列/按钮/筛选等）
2. 在开发页按语义查找对应元素
3. 找到后再框选

## No Position Projection Rule

禁止：把设计稿框直接映射到开发图同一位置

B端特别说明：B端页面列宽、内容长度差异使得位置投影误差远大于C端，必须严格独立定位。

# Design Completeness Rule

必须以设计稿为基准逐模块检查：

- 设计稿中每个模块 / 元素，都必须在开发页确认是否存在
- 若存在于设计稿但开发页不存在 → 必须输出正式问题
- 不得因为未发现差异而跳过模块
- **图表模块不得因"外形一致"而跳过内部结构检查**

---

# Data & Chart Exclusion Rule（B端）

以下差异默认不作为问题：
- 表格中的具体数值（金额、数量、百分比）
- 日期 / 时间字段的具体内容
- 用户名、头像、ID、排名、序号
- 图表数据值（柱高、折线点位、饼图比例）

**但以下属于设计稿固定定义，不属于动态数据，应正常上报：**
- 表头标签名称不同
- 筛选选项文案不同
- 操作按钮文案不同
- Tab 标签名称不同
- 列的增减或顺序变化
- 图表内部样式（折线实线/虚线、图例样式、网格线、坐标轴单位等）

---

# Problem Description Rule

只允许：
- 问题（包含两侧具体证据）
- 建议

不允许：
- 标题、位置、长分析
- reference 示例原句
- 截图里不存在的业务词
- 只描述差异类型而不说具体是什么（"不一致"、"缺失"、"不对"）

写法要求：
- 只写当前截图可见差异
- 句子必须可执行、可复核
- 不能因为 reference 出现过某个说法，就直接套到本次问题里

---

# Document Output Rules

结构固定：
1. 标题
2. 摘要
3. 页面总览
4. 问题表格

表头：序号 / 问题描述 / 状态 / 截图 / 日期 / 优先级 / 走查人 / 跟进人

---

# Restrictions

- 不得交换 dev/design
- 不得在未读图验证通过前输出结论
- 不得把动态数据当问题（数值、日期、用户名等）
- 不得把数据驱动图形当问题
- 不得混淆正式 / 疑似
- 不得框错对象
- 不得框空白
- 不得用位置映射代替元素匹配
- 不得把"数据差异"脑补为"规则问题"
- 不得复述 reference 示例中的现成问题句子
- 不得对图表只检查外形，必须检查内部结构
- 不得对文案差异只凭"感觉"下结论，必须逐字引用两侧实际文字
